# 六元围棋提子与终局计分修复计划

## Summary

修复 `assets/scripts/Game.js` 中两类规则错误：

- 提子 / 自杀禁入的死活判定错误：某些连通棋形会把仍然有气的活棋误判为死棋并提走。
- 终局按中国规则计分时的统计错误：地盘归属搜索不稳定，而且一整块空地目前只会加 1 分，不会按实际空地数量计分。

核心思路是把当前混杂的递归逻辑拆成清晰的“连通块搜索”和“空地区域搜索”两套遍历，分别服务于提子判定和终局计分。

## Current State Analysis

* 当前规则核心全部集中在 `assets/scripts/Game.js`。

* 用户已确认终局按中国规则处理，但仍要保留这个六边形棋盘“每点 3 邻接”的特殊性，不能直接套传统方格围棋的四邻接实现。

* 邻接关系由 `leftOf()`、`rightOf()`、`upDownOf()` 提供，符合该项目当前六边形棋盘建模方式。

* 死活判定入口是 `judgeDead()`，它会重置 `this.listTagNum`，再调用 `canPutChess(color, chess)` 判断指定棋块是否有气。

* `canPutChess()` 当前同时承担两件事：

  * 收集同色连通块到 `this.listTagNum`

  * 递归判断整块棋是否存在气

* 现有提子判定的关键问题：

  * 当递归再次走到“已经访问过的同色棋”时，函数直接返回 `false`

  * 上层用 `if (!b1 && !b2 && !b3) return false; else return true;` 聚合结果

  * 这意味着“走回已访问节点”会被当成一条失败分支，而不是“跳过已访问节点继续搜索”

* 在存在回环、分叉后汇合、或多路径连通的棋形中，这种写法会把同一块棋的内部连通重复访问错误地算成“无气证据”，从而导致：

  * 对方活棋被误提

  * 己方本可落子的点被误判为自杀

* 该问题与边界邻接数量无关，主要是搜索算法本身错误。

* `judgeWinner()` / `detectEnclosure()` 还存在额外的终局计分问题：

  * `detectEnclosure()` 对已访问空点直接返回 `false`，与死活判定里是同类错误，遇到回环或多路径空地区域时容易误判归属

  * `judgeWinner()` 遍历整个棋盘时，每次只在 `detectEnclosure()` 返回颜色时给 `cntRed` / `cntBlue` 加 1

  * 由于同一块空地的其它点会被 `this.listTagNum` 标记为已访问并跳过，当前实现实际上是“每块地只记 1 分”，而不是按空地数量计分

  * 当前调试残留 `console.log(13)` 不应保留在终局判定里

## Proposed Changes

### 1. 重构死活搜索函数

修改文件：`assets/scripts/Game.js`

* 删除当前 `canPutChess()` 中“已访问返回 false”的递归判定方式。

* 新增一个职责单一的遍历函数，建议命名为 `collectGroupAndCheckLiberty(color, chess)` 或含义等价的名称：

  * 输入：目标颜色、起始棋子

  * 输出：是否有气

  * 副作用：把该连通块的所有 `tagNum` 收集到 `this.listTagNum`

* 实现要求：

  * 仅在遇到空点时返回“有气”

  * 遇到异色棋时视为当前方向无气

  * 遇到已访问同色棋时，直接跳过，不得视为失败证据

  * 遍历完整个连通块后，只要任意相邻方向存在空点，就判定整块有气

* 实现方式可选 DFS 或显式栈遍历；优先选择更易读、更不易出错的写法。

### 2. 保持 `judgeDead()` / `removeDead()` 的接口语义稳定

修改文件：`assets/scripts/Game.js`

* `judgeDead()` 继续作为“判断指定颜色棋块是否死亡”的统一入口。

* 保留 `tiZou` 语义：

  * 为 `true` 时，确认为死棋后调用 `removeDead(color)`

  * 为 `false` 时，只做判定不提子

* 保证 `removeDead()` 仍然依赖 `this.listTagNum` 提子，这样可以最小范围接入新逻辑，不改动现有提子记录、劫、悔棋流程。

### 3. 校正己方落子合法性检查

修改文件：`assets/scripts/Game.js`

* `onTouchChess()` 在“未提到对方子”的分支里，会先把当前点暂设为己方颜色，再调用 `canPutChess(this.gameState)` 判断自杀禁入。

* 该流程在重构后需要改成调用新的“连通块有气”判定函数，确保：

  * 当前落子点作为己方新棋块起点参与完整搜索

  * 搜索前重置 `this.listTagNum`

  * 搜索结果只由真实气数决定，不受重复访问影响

### 4. 提升可读性，删除旧的混合职责写法

修改文件：`assets/scripts/Game.js`

* 删除旧版 `canPutChess()` 的混合职责实现，避免旧逻辑和新逻辑并存。

* 增加一个统一的邻点获取函数，建议命名为 `getNeighbors(chess)` 或含义等价的名称，内部继续复用 `upDownOf()`、`leftOf()`、`rightOf()`，减少各处重复三连调用。

* 注释统一改成说明“连通块”“气”“已访问节点跳过”等真实规则含义，便于后续维护。

### 5. 重写终局地盘判定，按中国规则统计

修改文件：`assets/scripts/Game.js`

* 保留中国规则口径：

  * 存活棋子本身计分

  * 被单方包围的空地按空点数量计分

  * 同时接触双方或无明确归属的空地不计分

* 删除当前 `detectEnclosure()` “递归返回单个颜色字符串”的写法，改成更明确的空地区域遍历，建议命名为 `collectEmptyRegion()` / `getRegionOwner()` 或含义等价的名称。

* 新的空地区域遍历需要一次性得到：

  * 当前空地区域包含的全部点位

  * 该区域边界接触到的颜色集合

* 归属判定规则：

  * 只接触 `red`，则该区域归 `red`

  * 只接触 `blue`，则该区域归 `blue`

  * 同时接触两方，则为公共地，不计分

* `judgeWinner()` 重写为显式累计：

  * 遍历每个棋点

  * 红蓝棋子各自直接计 1 分

  * 遇到未访问空点时，整块区域只处理一次，并按区域大小一次性加分

* 删除 `cntNull` 和 `console.log(13)` 这类无效或调试残留代码。

### 6. 保持终局触发流程不变，但让结果可信

修改文件：`assets/scripts/Game.js`

* `tingyishou()` 仍保持“双方都停一手后进入终局结算”的现有交互。

* `gameOver()` / `judgeWinner()` 的调用关系保持不变，只替换内部计分实现，避免影响 UI 与现有流程。

* 不在本次改动中额外引入“终局标记死子”交互；本次只修复当前代码里明确存在的统计与遍历错误。

## Assumptions & Decisions

* 不改动棋盘几何规则；`leftOf()`、`rightOf()`、`upDownOf()` 继续作为唯一邻接来源。

* 不引入兼容旧逻辑的分支；直接移除错误判定实现，保持代码整洁。

* 终局按中国规则实现：棋子本身和明确归属的空地都计分。

* 不把六边形棋盘强行改写成传统围棋四邻接或正方形棋盘逻辑。

* 不在本次改动中扩展更复杂的争议死活裁定机制，只修复代码中已确认的遍历和计分错误。

* 不新增复杂基础设施；优先在现有 `Game.js` 内完成小范围、清晰的重构。

## Verification Steps

实施后按以下顺序验证：

1. 基础提子

   * 构造单子仅剩 1 口气的局面

   * 最后一口气被堵住后应正确提子

2. 活棋不被误提

   * 构造一块通过另一条分支仍保留气的连通棋形

   * 即使搜索过程中出现回到已访问节点，也不得被提走

3. 环形/回路棋形

   * 构造存在回环的同色连通块

   * 只要整块任意位置仍有气，`judgeDead()` 必须返回存活

4. 自杀禁入

   * 构造“看似被围但能联通己方外部气”的落点

   * 应允许落子

   * 构造真正无气且不能提子的落点

   * 应禁止落子

5. 终局中国规则计分

   * 构造被单方完整包围的多格空地区域

   * 结算时应按空地区域实际大小累计分数，而不是只记 1 分

   * 构造同时接触红蓝双方的公共地

   * 结算时不应给任一方加分

   * 确认存活棋子本身仍计分

6. 回归检查

   * 单子提子、多子提子、劫点记录、悔棋恢复流程保持原行为

   * 对 `assets/scripts/Game.js` 运行语法/诊断检查，确认无新错误
